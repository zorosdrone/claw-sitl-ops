# 導入手順 / テスト手順

このドキュメントは、OpenClaw 最新版の環境に本プロジェクトを導入し、ArduPilot SITL を操作できる状態まで持っていくための手順書です。

想定構成の全体像は [docs/11_想定構成図.md](11_想定構成図.md) を参照してください。

前提:
- OpenClaw は導入済みで、バージョンは最新版
- XServer の無料 VPN により TailScale は導入済み
- OpenClaw を動かしているホストは Linux シェルを使える
- ArduPilot SITL を起動する対象ホスト上で Python 3 と bash が使える

## 1. このプロジェクトの役割

このリポジトリは、OpenClaw から ArduPilot SITL を MAVLink 経由で操作する Skill 一式です。

主に次の操作を提供します。

- SITL の起動 / 停止
- 機体状態の取得
- arm
- takeoff
- mode 変更
- param get / set

実体は次のファイルです。

- [SKILL.md](../SKILL.md)
- [scripts/sitl_mav.py](../scripts/sitl_mav.py)
- [scripts/sitl_dispatch.sh](../scripts/sitl_dispatch.sh)
- [scripts/setup_venv.sh](../scripts/setup_venv.sh)

## 2. 導入前に確認すること

最初に、OpenClaw 実行ホストで以下を確認してください。

1. OpenClaw が起動できること
2. TailScale 経由で必要なネットワーク到達性があること
3. ArduPilot のソース一式が存在すること
4. Python 3 が使えること
5. bash が使えること

確認コマンド例:

```bash
python3 --version
bash --version
```

ArduPilot の配置先は、既定では次を想定しています。

```bash
~/.openclaw/workspace/GitHub/ardupilot
```

もし別の場所に ArduPilot を置いている場合は、後述の環境変数 `SITL_AP_ROOT` で上書きします。

## 3. Skill を OpenClaw の skills 配下へ配置する

このリポジトリは、そのまま任意の場所に置くだけではなく、OpenClaw の skills 配下に `sitl-ops` というディレクトリ名で配置する前提です。

理由:
- [SKILL.md](../SKILL.md) 内の実行例が `skills/sitl-ops/...` を前提にしている
- [scripts/sitl_dispatch.sh](../scripts/sitl_dispatch.sh) と [scripts/setup_venv.sh](../scripts/setup_venv.sh) は、skills 配下に配置された構成を前提にルートを解決する

配置例:

```bash
mkdir -p ~/.openclaw/workspace/skills
cp -r /path/to/claw-sitl-ops ~/.openclaw/workspace/skills/sitl-ops
```

配置後の想定構成:

```text
~/.openclaw/workspace/
	skills/
		sitl-ops/
			SKILL.md
			scripts/
				sitl_mav.py
				sitl_dispatch.sh
				setup_venv.sh
```

注意:
- ディレクトリ名は `sitl-ops` に揃えてください
- 既に同名ディレクトリがある場合は、上書き前に退避してください

## 4. Python 仮想環境を作成する

次に、Skill 用の Python 仮想環境を作成します。

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/setup_venv.sh
```

このスクリプトで実施される内容:
- `.venv` の作成
- `pip` の更新
- `pymavlink` のインストール
- `empy==3.3.4` のインストール
- `MAVProxy` のインストール

成功時の想定出力:

```text
OK: .venv ready with pymavlink + empy + MAVProxy
Use: export SITL_VENV_PYTHON=.../.venv/bin/python
```

## 5. 必要に応じて環境変数を設定する

既定値で運用できる場合はこの章を飛ばして構いません。

毎回 `export` したくない場合は、ローカル構成用テンプレートから `.sitl-ops.env` を作成できます。

```bash
cd ~/.openclaw/workspace
cp skills/sitl-ops/sitl-ops.local.env.example .sitl-ops.env
```

このファイルは [scripts/sitl_dispatch.sh](../scripts/sitl_dispatch.sh) から自動読込されます。

ArduPilot の場所や仮想環境の場所が既定と異なる場合は、SITL 起動前に次を設定します。

```bash
export SITL_VENV_PYTHON="$HOME/.openclaw/workspace/.venv/bin/python"
export SITL_AP_ROOT="$HOME/.openclaw/workspace/GitHub/ardupilot"
export SITL_AP_VENV_ACTIVATE="$HOME/venv-ardupilot/bin/activate"
export SITL_START_ARGS="-v Copter -L Kawachi --no-mavproxy"
export SITL_LOG="/tmp/sitl_copter.log"
```

各変数の意味:
- `SITL_VENV_PYTHON`: Skill 側 Python 実行ファイル
- `SITL_AP_ROOT`: ArduPilot リポジトリのルート
- `SITL_AP_VENV_ACTIVATE`: ArduPilot 側の仮想環境 activate スクリプト
- `SITL_START_ARGS`: `sim_vehicle.py` 起動引数
- `SITL_LOG`: SITL 起動ログ出力先

設定値を確認する場合:

```bash
echo "$SITL_VENV_PYTHON"
echo "$SITL_AP_ROOT"
echo "$SITL_AP_VENV_ACTIVATE"
echo "$SITL_START_ARGS"
echo "$SITL_LOG"
```

## 6. コマンドが呼べることを確認する

まず、Skill 本体のヘルプが出ることを確認します。

```bash
cd ~/.openclaw/workspace
.venv/bin/python skills/sitl-ops/scripts/sitl_mav.py --help
```

次に、Discord 風のコマンドを受ける dispatcher が動くことを確認します。

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

この段階では SITL 未起動なら heartbeat 待ちで失敗しても問題ありません。重要なのは、スクリプト自体が見つかり、Python 依存関係のエラーで落ちないことです。

## 7. テスト手順

ここからは、導入後の動作確認を順番に行います。最小限の E2E 確認として、次の順でテストしてください。

### 7-1. テスト 1: SITL を起動する

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl start"
```

期待結果:
- `ok: true` が返る
- `status` が `started` または `already_running` になる
- ログファイルのパスが返る

起動に失敗した場合はログを確認します。

```bash
tail -n 120 /tmp/sitl_copter.log
```

### 7-2. テスト 2: 機体状態を取得する

SITL 起動後、heartbeat が安定するまで数秒待ってから実行します。

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

期待結果:
- JSON で結果が返る
- `mode` が取得できる
- `armed` が `false` である
- `position` が取得できる
- `map_url` が含まれる

このテストで失敗する場合は、MAVLink 接続先の解決に失敗している可能性があります。必要なら `SITL_MASTER` を明示してください。

例:

```bash
export SITL_MASTER="udp:127.0.0.1:14550"
```

### 7-3. テスト 3: arm を実行する

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl arm"
```

期待結果:
- `ok: true`
- `action: arm`
- `status.armed: true`

続けて状態確認:

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

### 7-4. テスト 4: takeoff を実行する

10m の離陸テスト例:

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl takeoff 10"
```

期待結果:
- `ok: true`
- `action: takeoff`
- `target_alt_m: 10`

このコマンドは内部で `GUIDED` へ切り替え、必要なら arm した上で `MAV_CMD_NAV_TAKEOFF` を送ります。

コマンド実行後、3 から 5 秒待って状態を確認します。

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

確認ポイント:
- `mode` が `GUIDED` になっていること
- `armed` が `true` のままであること
- `position.relative_alt_m` が上昇していること

### 7-5. テスト 5: mode を変更する

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl mode GUIDED"
```

期待結果:
- `ok: true`
- `action: mode`
- `mode: GUIDED`

別モードを試す場合は、SITL 側が受け付けるモード名を使ってください。

### 7-6. テスト 6: param get を実行する

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl param get ARMING_CHECK"
```

期待結果:
- `ok: true`
- `action: param-get`
- `name: ARMING_CHECK`
- `value` が返る

### 7-7. テスト 7: param set を実行する

`param set` は値を書き換えるため、必ず元の値を控えてから実行してください。

1. まず現在値を取得する

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl param get ARMING_CHECK"
```

2. テスト値を設定する

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl param set ARMING_CHECK 1"
```

3. 値が反映されたことを確認する

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl param get ARMING_CHECK"
```

4. 必要なら元の値に戻す

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl param set ARMING_CHECK <元の値>"
```

注意:
- 共有環境での param 書き換えは他の検証に影響するため、使い捨ての SITL でのみ実施してください

### 7-8. テスト 8: SITL を停止する

最後にプロセスを停止します。

```bash
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl stop"
```

期待結果:
- `ok: true`
- `action: stop`
- `status: stopped`

## 8. OpenClaw から利用する際の確認ポイント

OpenClaw からこの Skill を呼ぶ場合は、少なくとも次を確認してください。

1. Skill が OpenClaw の skills パス配下に存在すること
2. OpenClaw から [SKILL.md](../SKILL.md) を参照できること
3. 実行ホスト上で `bash` と `python3` が使えること
4. `.venv` が作成済みであること
5. ArduPilot と MAVLink 接続先が正しいこと

Discord 風コマンドの例:

```text
!sitl start
!sitl status
!sitl arm
!sitl takeoff 10
!sitl mode GUIDED
!sitl param get ARMING_CHECK
!sitl param set ARMING_CHECK 1
!sitl stop
```

## 9. よくある失敗と対処

### heartbeat が返らない

症状:
- `No heartbeat from vehicle`

対処:
1. SITL が起動しているか確認する
2. `!sitl status` 実行前に数秒待つ
3. `SITL_MASTER` を明示する
4. `mavproxy.py` または `sim_vehicle.py` の起動状態を確認する

### ArduPilot の配置先が違う

症状:
- `AP_ROOT not found` が返る

対処:

```bash
export SITL_AP_ROOT="<実際の ardupilot のパス>"
```

### Python 依存関係が足りない

症状:
- `ModuleNotFoundError`
- `you need to install empy`
- `mavproxy.py` が見つからない

対処:

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/setup_venv.sh
```

## 10. 導入完了の判定基準

以下をすべて満たせば、導入完了です。

1. `setup_venv.sh` が成功する
2. `!sitl start` が成功する
3. `!sitl status` で JSON の状態が返る
4. `!sitl arm` が成功する
5. `!sitl takeoff 10` 後に高度上昇を確認できる
6. `!sitl param get` が成功する
7. `!sitl stop` が成功する

この状態まで確認できれば、OpenClaw からの SITL 基本操作は通ったと判断して構いません。
