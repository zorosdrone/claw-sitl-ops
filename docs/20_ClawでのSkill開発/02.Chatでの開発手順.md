# 開発手順

このドキュメントは、[README.md](../README.md) に書かれている開発経緯を、実際に再現しやすい手順へ分解したものです。

対象:
- OpenClaw から ArduPilot SITL を操作する Skill を新規に作る場合
- 既存の Skill を、実運用しながら段階的に育てたい場合
- `status` / `arm` / `takeoff` / `mode` / `param get/set` を最小構成から安定化したい場合

この手順は、次の考え方で進めます。

- 最初から全部入りにしない
- まずは最短の E2E を通す
- 実運用で失敗した箇所を少しずつ直す
- 返却は人間向け文章より JSON を優先する

## 1. 最初に決めること

開発に入る前に、Skill の責務を絞ります。

この Skill の責務:
- ArduPilot SITL へ MAVLink で接続する
- OpenClaw から呼びやすい形でコマンドを受ける
- 結果を JSON で返す

最初の段階で対象にするコマンド:
- `status`
- `arm`
- `disarm`
- `takeoff`
- `mode`
- `param get`
- `param set`

この段階でまだ後回しにしてよいもの:
- 高度なミッション管理
- 長時間の自動監視
- 複雑な対話フロー
- 見た目の整った自然文整形

重要なのは、最初から広げ過ぎず、`コマンドを受ける -> MAVLink を送る -> 結果を返す` の一本線を成立させることです。

## 2. 最小ファイル構成を作る

README にある通り、最初に必要なファイルは次の 4 つです。

- [SKILL.md](../SKILL.md)
- [scripts/sitl_mav.py](../scripts/sitl_mav.py)
- [scripts/sitl_dispatch.sh](../scripts/sitl_dispatch.sh)
- [scripts/setup_venv.sh](../scripts/setup_venv.sh)

補助ファイルとして、環境差分を吸収するためのサンプル env も用意します。

- [sitl-ops.local.env.example](../sitl-ops.local.env.example)
- [sitl-ops.remote.env.example](../sitl-ops.remote.env.example)

役割は次のとおりです。

1. [SKILL.md](../SKILL.md)
	OpenClaw に対して、この Skill が何をし、どう呼ばれるかを定義します。

2. [scripts/sitl_mav.py](../scripts/sitl_mav.py)
	実際の MAVLink 操作を担当します。

3. [scripts/sitl_dispatch.sh](../scripts/sitl_dispatch.sh)
	`!sitl status` のような入力を解釈し、Python スクリプトへ橋渡しします。

4. [scripts/setup_venv.sh](../scripts/setup_venv.sh)
	必要な Python 実行環境をまとめて準備します。

この構成にしておくと、仕様変更時に責務が分離され、修正箇所が追いやすくなります。

## 3. OpenClaw TUI で最初の雛形を作る

README にあるとおり、最初の実装は `openclaw tui` を使って対話的に生成しています。

### 3-1. 初期プロンプトを決める

最初の生成で重要なのは、欲しい機能と必要ファイルを過不足なく伝えることです。

ここで曖昧にしない方がよい項目:

- Skill 名
- 配置先ディレクトリ
- `SKILL.md` の frontmatter の `name`

この 3 つを省くと、TUI が雛形を作れても、あとでフォルダ名と Skill 名がずれやすくなります。

このリポジトリの運用では、次を揃える前提にします。

- フォルダ名: `skills/sitl-ops/`
- Skill 名: `sitl-ops`
- [SKILL.md](../SKILL.md) の frontmatter: `name: sitl-ops`

再現用の初期プロンプト:

```text
ArduPilot SITL を OpenClaw から操作する Skill を作って。
Skill 名は sitl-ops。
配置先は skills/sitl-ops/。
SKILL.md の frontmatter の name は sitl-ops にして。

MAVLink 経由で以下を実行できるようにして:
- status
- arm/disarm
- takeoff
- mode 変更
- param get/set

必要ファイル:
- SKILL.md
- scripts/sitl_mav.py
- scripts/sitl_dispatch.sh
- scripts/setup_venv.sh

出力は Discord で扱いやすいよう JSON で返して。
まずは最小構成で動くところまで作って。
```

このように書いておくと、TUI が生成した内容をそのまま `skills/sitl-ops/` へ置きやすくなります。

逆に、Skill 名を指定しない場合は次の点がぶれます。

- フォルダ名
- `SKILL.md` の `name`
- 後続ドキュメント内のコマンド例
- `skills/<skill_name>/...` 前提の配置手順

### 3-2. 最初の生成で見るべきポイント

雛形が出たら、次だけを確認します。

1. コマンド入口があるか
2. Python 側に `status` / `arm` / `takeoff` などの分岐があるか
3. 返却が JSON になっているか
4. 外部依存の準備手順が明示されているか

この時点では、実装が多少粗くても問題ありません。
最優先は「呼べること」と「返せること」です。

## 4. 最短 E2E を成立させる

雛形ができたら、すぐに全コマンドを作り込まず、次の順で E2E を通します。

### 4-1. 仮想環境を作る

まず依存関係が入ることを確認します。
OpenClaw workspace 配下へ配置している場合の実行例:

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/setup_venv.sh
```

ここで見たいこと:
- `.venv` が作成されること
- `pymavlink` の導入で落ちないこと
- 実行完了メッセージが出ること

補助確認コマンド:

```bash
ls skills/sitl-ops/.venv/bin/python
skills/sitl-ops/.venv/bin/python --version
```

ここで失敗する場合は、MAVLink 以前に環境の問題です。先に解消します。

### 4-2. `status` だけ先に通す

最初に通すべきコマンドは `status` です。

理由:
- 破壊的でない
- 接続確認になる
- モード、位置、arm 状態など複数情報をまとめて見られる

確認したいこと:
- heartbeat が取れるか
- JSON が返るか
- 位置情報が読めるか
- エラー時も JSON で返るか

最小確認コマンド:

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

dispatcher を通さず Python 単体で確認する例:

```bash
cd ~/.openclaw/workspace
skills/sitl-ops/.venv/bin/python skills/sitl-ops/scripts/sitl_mav.py status --target udp:127.0.0.1:14550
```

期待する出力イメージ:

```json
{"mode":"GUIDED","armed":false,"battery_remaining_pct":100,"position":{"lat":35.0,"lon":139.0,"relative_alt_m":0.0},"attitude_deg":{"roll":0.0,"pitch":0.0,"yaw":90.0},"map_url":"https://maps.google.com/?q=35.0,139.0"}
```

SITL 未起動時の失敗確認も先にやっておくと、正常系と異常系の差が分かりやすくなります。

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

失敗時は、たとえば次のような JSON を返せれば十分です。

```json
{"ok":false,"error":"No heartbeat from vehicle"}
```

### 4-3. 次に `arm` を通す

`status` が通ったら `arm` を実装します。

確認したいこと:
- コマンド送信後に armed 状態へ遷移するか
- 応答待ちのタイミングが短すぎないか
- arm 成功判定が雑すぎないか

確認コマンド:

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl arm"
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

期待する見え方:

```json
{"ok":true,"action":"arm","status":{"mode":"GUIDED","armed":true}}
```

arm 後の `status` では、少なくとも `armed: true` が確認できることを見ます。

### 4-4. 最後に `takeoff` を通す

`takeoff` は状態依存が強く、失敗要因が増えるため後に回します。

確認したいこと:
- 必要なら `GUIDED` へ移れるか
- 必要なら arm を先行できるか
- 離陸コマンド送信後に高度上昇が確認できるか

確認コマンド:

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl takeoff 10"
sleep 3
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

期待する見え方:

```json
{"ok":true,"action":"takeoff","target_alt_m":10.0,"status":{"mode":"GUIDED","armed":true}}
```

確認の焦点:
- `action` が `takeoff` で返ること
- `target_alt_m` が入力値と一致すること
- 続く `status` で `relative_alt_m` が増えること

ここまで通れば、README にある「最短 E2E」は成立です。

## 5. 返却形式を最初から JSON に揃える

README でも触れている通り、この Skill は Discord などの外側で扱いやすいことが重要です。

そのため、初期段階から返却形式は JSON に揃えます。

最低限含めたい項目の例:

- `ok`
- `action`
- `mode`
- `armed`
- `position`
- `message`
- `error`

方針:
- 成功時も失敗時も JSON を返す
- 人間向けの長文説明は外側で付ける
- スクリプト側は機械可読性を優先する

このルールにしておくと、後で Discord、CLI、Web UI など複数の呼び出し元へ展開しやすくなります。

## 6. Discord で実運用しながら改善する

README にある育成プロセスの本体はここです。

TUI で作った後は、Discord 上で実際にコマンドを叩きながら改善を回します。

基本ループ:

1. 実コマンドを投げる
2. 成功か失敗かを観測する
3. 失敗理由を切り分ける
4. Skill またはスクリプトを修正する
5. 同じコマンドで再テストする
6. 小さくコミットする

実際の入力例:

```text
!sitl status
!sitl arm
!sitl takeoff 10
!sitl mode GUIDED
```

ローカルシェルから再現する場合の例:

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl mode GUIDED"
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl arm"
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl takeoff 10"
```

パラメータ確認まで含める場合:

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl param get ARMING_CHECK"
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl param set ARMING_CHECK 1"
```

このループでは、机上で考えた仕様より、運用時の引っかかりを優先して潰します。

## 7. `status` の UX を先に改善する

README にある改善の一つが、`!sitl status` への Google Maps URL 追加です。

これは小さな変更に見えますが、運用ではかなり効きます。

### 7-1. なぜ先に `status` を改善するのか

理由:
- 一番よく叩くコマンドだから
- 失敗解析の入口になるから
- 操作前後の確認にも使うから

### 7-2. 追加する内容

`status` の戻り値に次を含めます。

- 緯度
- 経度
- 相対高度
- モード
- arm 状態
- `map_url`

追加後の確認コマンド:

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

出力イメージ:

```json
{"mode":"GUIDED","armed":true,"position":{"lat":35.0,"lon":139.0,"relative_alt_m":12.4},"map_url":"https://maps.google.com/?q=35.0,139.0"}
```

### 7-3. 改善後に確認すること

確認項目:
- `status` 実行だけで位置確認できるか
- オペレータが緯度経度をコピペせずに済むか
- 戻り値 JSON が肥大化し過ぎていないか

要点は、情報量を増やしつつ、判断までの手数を減らすことです。

## 8. 離陸フローは arm/disarm レースを疑う

README にある「`NAV_TAKEOFF` 中心に統一した」という改善は、実運用でよく起きる問題への対処です。

### 8-1. ありがちな失敗

離陸系で起きやすい問題:
- arm が終わる前に takeoff を送っている
- モード切替の反映前に次のコマンドを送っている
- 離陸手順が環境ごとに微妙に違う

### 8-2. 修正方針

手動で複数コマンドを組み合わせるより、Python 側の `takeoff` 処理で次をまとめて行う方が安定します。

1. 接続確認
2. 必要なら `GUIDED` へ変更
3. 必要なら arm 実行
4. `MAV_CMD_NAV_TAKEOFF` を送信
5. 上昇確認用の状態を返す

### 8-3. 確認手順

```text
!sitl takeoff 10
!sitl status
```

ローカルでの再現コマンド:

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl mode GUIDED"
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl takeoff 10"
sleep 3
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

切り分け用に `mode` 単体も確認します。

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl mode GUIDED"
```

見るべき点:
- `armed` が維持されるか
- `mode` が想定どおりか
- `relative_alt_m` が上がるか

単にコマンドが成功したと出るだけでなく、状態遷移が一貫しているかまで見る必要があります。

## 9. ミッション系ワークフローは小さい形から検証する

README では、正方形ミッションや星形ミッションの upload と実行まで確認しています。

ここでは、いきなり複雑な実運用をせず、次の順で増やします。

### 9-1. まず単純な形で waypoint 群を作る

最初の候補:
- 正方形
- 星形

理由:
- 目視で異常を見つけやすい
- waypoint 順序の不整合に気付きやすい
- 実行結果を地図上で判断しやすい

### 9-2. upload だけを先に確認する

最初に確認すること:
- waypoint 数が想定どおりか
- upload が最後まで通るか
- mission ack が正常か

このリポジトリの現時点の最小コマンド集合には mission upload はまだ入っていないため、この段階では次のどちらかで検証を進めます。

1. `scripts/sitl_mav.py` に upload 用サブコマンドを追加してから試す
2. 別スクリプトで waypoint upload を行い、その後 `status` と `mode` だけ本 Skill で確認する

追加実装前にやるべきこと:
- waypoint 定義の入力形式を決める
- 成功時 JSON に何を返すか決める
- upload 失敗時の error JSON を決める

### 9-3. 次に `AUTO` で開始する

upload が通ったら、開始フェーズを確認します。

既に `mode` コマンド自体はあるため、AUTO への遷移確認だけ先にできます。

```bash
cd ~/.openclaw/workspace
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl mode AUTO"
bash skills/sitl-ops/scripts/sitl_dispatch.sh "!sitl status"
```

確認項目:
- `AUTO` へ遷移するか
- ミッション進行が始まるか
- 実行中に status で追跡できるか

### 9-4. 実行後の報告まで確認する

README の通り、実行後テレメトリや状態報告まで見ます。

ここで見ること:
- 実行完了後に状態が破綻していないか
- 失敗時のメッセージが足りるか
- オペレータが次に何を見るべきか分かるか

## 10. 定期報告は運用で必要になってから絞る

README にある「定期ステータス報告の開始/停止や出力内容の調整」は、初期実装で全部決め切らない方がよい部分です。

理由:
- 運用者ごとに欲しい頻度が違う
- 欲しい項目も場面で変わる
- 出し過ぎるとノイズになる

設計方針:
- 開始コマンドを用意する
- 停止コマンドを用意する
- 最初は必要最小限の項目だけ出す
- フィードバックを見て粒度を増減する

## 11. 1 回の修正を小さく保つ

README にある「小さくコミット」は重要です。

一度に多くを変えると、どの修正で安定したのか追えなくなります。

1 回の修正単位の例:
- `status` に `map_url` を追加するだけ
- `takeoff` を `NAV_TAKEOFF` ベースへ変えるだけ
- mission upload のエラー JSON を増やすだけ

避けたい進め方:
- 返却形式変更
- コマンド体系変更
- ミッション処理変更
- 定期通知追加

これらを一度にまとめて変えることです。

## 12. 最後に README へ要点を戻す

詳細化した内容は、この [docs/21.開発手順.md](21.開発手順.md) に置き、README には要点だけを残すのが扱いやすい構成です。

README に残す内容:
- この Skill が何をするか
- どのドキュメントを見ればよいか
- 開発の要約

詳細ドキュメントに寄せる内容:
- 何から実装するか
- どういう順でテストするか
- どこをどう改善したか
- 実運用で何を見て直すか

## 13. 再現用チェックリスト

この README の内容を再現したい場合は、次の順で進めます。

1. Skill の責務を `SITL 操作 + JSON 返却` に絞る
2. [SKILL.md](../SKILL.md)、[scripts/sitl_mav.py](../scripts/sitl_mav.py)、[scripts/sitl_dispatch.sh](../scripts/sitl_dispatch.sh)、[scripts/setup_venv.sh](../scripts/setup_venv.sh) を作る
3. `status` を最初に通す
4. `arm` を通す
5. `takeoff` を通す
6. Discord から実コマンドで試す
7. `status` の見やすさを改善する
8. 離陸フローを安定化する
9. ミッション upload と `AUTO` を小さく検証する
10. 必要になった機能だけ追加する

この順番を守ると、README にある開発経緯を、作業手順として無理なくたどれます。
